1. The Problem

Classification

Will an ICE stop be delayed ≥ 6 minutes?

Binary output: on-time or delayed. The 6-minute threshold matches Deutsche Bahn's definition of a "significant delay."

Primary metrics: F1, ROC-AUC — accuracy alone is misleading under class imbalance.

Regression

How many minutes late will it be?

Continuous prediction. Values range from -5 (early) to 298 minutes late. MAE is directly interpretable in minutes.

Primary metrics: MAE (interpretable) and RMSE (penalizes severe delays).

Why both? Operators need to know whether to trigger mitigations and how much delay to communicate. Three research questions guide the project: RQ1 — can operational features alone predict delays? RQ2 — does adding weather improve predictions? RQ3 — which weather variables matter most?

3. Data Cleaning

Four categories of cleaning decisions:

Canceled stops

11,497 rows removed (7.28%). Cannot predict delay for trips that never ran. Removed before any analysis.

Null station names

Filled from xml_station_name column. Station identity is important for modeling, so dropping would lose information.

Target leakage excluded

arrival_change_time and departure_change_time encode the actual delay. Including them would make the model trivially perfect but useless for real prediction.

Redundant columns dropped

rain: perfectly correlated with precipitation (r=1.0).
snowfall: zero variance in July (all 0s). Both are dropped in feature engineering.

Challenge: All 5 timestamp columns were naive (no timezone). Standardized to Europe/Berlin using pandas.tz_localize. 14,121 rows without departure_planned_time flagged as non-mergeable.

5. Baselines & Metrics

Every model must beat a naive baseline before we consider it useful.

Classification

Naive baseline: Always predict "on-time" → 57.7% accuracy

Primary metrics: F1-score and ROC-AUC.

Accuracy is misleading here. A model scoring 60% accuracy is barely above the naive baseline. F1 balances precision and recall; ROC-AUC evaluates performance across all thresholds.

Regression

Naive baseline: Always predict mean delay (10.57 min) → MAE ~13.5 min

Primary metrics: MAE and RMSE.

MAE is interpretable in minutes for passengers. RMSE penalizes large errors (severe delays that most affect operations). R² as secondary.

Evaluation protocol: Time-based (chronological) train/test split — no random shuffle, to prevent temporal leakage. Ablation study: same algorithm trained with and without weather features to directly answer RQ2 (does weather improve prediction?).

Key Design Decisions

6 min

Delay threshold

Aligned with DB's "significant delay." 5 min is too noisy (routine variation), 15 min yields too few positive cases.

Time split

Train/test split

Random split leaks future data into training. Chronological split reflects real deployment: predict future from past.

July only

Project scope

Single-month proof-of-concept for fast iteration. Pipeline is parameterized by date — extensible to any month.

ICE only

Train type filter

ICE is high-profile long-distance. Removing other train types avoids confounding from different operational profiles.

Planned time

Merge key

departure_planned_time (not actual) used for weather join. Actual times encode the delay — that would leak the target.

LEFT JOIN

Merge type

Preserves all 146K ICE rows. An inner join would silently drop 14K rows without weather, hiding data quality issues.

7. Code Review

Pipeline Architecture

Five sequential Jupyter notebooks, each outputs a Parquet file consumed by the next:

01 Define → 02 Acquire → 03 Clean → 04 Integrate → 05 Explore

Each notebook loads configuration from data/reference/*.json and writes a structured JSON report upon completion. No hardcoded paths, no manual steps.

What Makes It Special

  • Config-driven: 21 JSON files define data dictionaries, merge strategies, cleaning rules, and per-notebook reports — full audit trail without a database.
  • Streaming Parquet: The full DB is 54.8 GB. Notebooks read in 100K-row chunks via PyArrow, filter to ICE only, and output a 5.2 MB subset.
  • Auto-generated reports: Every notebook writes a structured JSON summary (cleaning stats, merge validation, EDA findings). Rerun any notebook to regenerate.
  • Decoupled presentation: Slides are a standalone HTML file in a separate repository, deployable via Netlify without touching the analysis pipeline.
