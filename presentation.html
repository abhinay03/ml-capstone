1. The Problem 2 / 9

What we predict

Classification

Will an ICE stop be delayed ≥ 6 min?

Binary: on-time or delayed. Threshold matches Deutsche Bahn's definition of a "significant delay."

Primary metrics: F1, ROC-AUC

Regression

How many minutes late?

Continuous: -5 (early) to 298 minutes late. MAE is directly interpretable in minutes.

Primary metrics: MAE, RMSE

Why both? Operators need to know whether to trigger mitigations and how much delay to communicate. Three research questions: RQ1 (operational features only), RQ2 (does weather improve prediction?), RQ3 (which weather variables matter?).

3. Data Cleaning 4 / 9

Canceled stops removed

11,497 rows (7.28%). You cannot predict delay for trips that never ran.

Null station names filled

Filled from xml_station_name. Dropping would lose station identity, a key feature.

Target leakage excluded

arrival_change_time and departure_change_time encode the actual delay — including them would leak the target.

Redundant columns dropped

rain: r=1.0 with precipitation. snowfall: zero variance in July (all 0s).

Challenge: All 5 timestamp columns were naive (no timezone). Standardized to Europe/Berlin. 14,121 rows without departure_planned_time flagged as non-mergeable.

5. Baselines & Metrics 6 / 9

Every model must beat a naive baseline before we consider it useful.

Classification

Naive baseline: Always predict "on-time" → 57.7% accuracy

Primary: F1-score and ROC-AUC.

Accuracy is misleading. A model at 60% accuracy is barely above naive. F1 balances precision and recall; ROC-AUC evaluates across all thresholds.

Regression

Naive baseline: Always predict mean (10.57 min) → MAE ~13.5 min

Primary: MAE and RMSE.

MAE interpretable in minutes. RMSE penalizes large errors (severe delays). R² as secondary.

Evaluation: Time-based (chronological) split — prevents temporal leakage. Ablation study: same model with vs without weather features → answers RQ2.
Key Design Decisions 7 / 9

6 min

Delay threshold

Aligned with DB's "significant delay." 5 min too noisy, 15 min too few cases.

Time split

Train/test split

Chronological, not random. Random leaks future into past. Mirrors real deployment.

July only

Project scope

Single-month proof-of-concept. Pipeline parameterized by date for easy extension.

ICE only

Train type

High-profile long-distance. No confounding from different operational profiles.

Planned time

Merge column

Used departure_planned_time. Actual times encode the delay — target leakage.

LEFT JOIN

Merge type

Preserves all rows. Inner join silently drops 14K rows without weather.

7. Code Review 9 / 9

Pipeline Architecture

Five sequential Jupyter notebooks, each outputs a Parquet file for the next:

01 Define → 02 Acquire → 03 Clean → 04 Integrate → 05 Explore

Each reads config from data/reference/*.json and writes a JSON report. No hardcoded paths. Fully reproducible.

What's Special

  • Config-driven: 21 JSON files define data dictionaries, merge strategies, cleaning rules, per-notebook reports. Full audit trail without a database.
  • Streaming Parquet: 54.8 GB DB read in 100K-row chunks via PyArrow. Output: 5.2 MB ICE subset.
  • Auto-reports: Every notebook writes a structured JSON summary. Rerun any notebook to regenerate.
  • Decoupled slides: Standalone HTML in separate repo, deployable via Netlify.
