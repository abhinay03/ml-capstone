1 — The Problem

Classification

Will this stop be delayed ≥ 6 min?

Binary: on-time or delayed. Threshold matches Deutsche Bahn's "significant delay" definition.

Metric: F1 & ROC-AUC (accuracy is misleading under imbalance)

Regression

How many minutes late?

Continuous prediction from −5 (early) up to 298 min. MAE is interpretable in minutes for passengers.

Metric: MAE (interpretable) & RMSE (penalizes severe delays)

Why both? Operators need to know whether to trigger mitigations and how much to inform passengers. Three research questions: RQ1 — can operational features alone predict? RQ2 — does weather improve prediction? RQ3 — which weather variables matter most?

3 — Cleaning

Four categories of cleaning decisions:

DROP Canceled stops

11,497 rows (7.28%) removed. You cannot predict delay for trips that never ran.

FILL Null station names

Filled from xml_station_name. Chosen over dropping because station identity is a key feature.

EXCLUDE Target leakage

arrival_change_time & departure_change_time encode the actual delay. Using them would leak the target.

DROP Redundant columns

rain (r=1.0 with precipitation), snowfall (zero variance in July).

Challenge: Timestamps were naive (no timezone). Standardized all 5 timestamp columns to Europe/Berlin. 14,121 rows without departure_planned_time flagged as non-mergeable.

5 — Baselines & Metrics

We must beat naive baselines before introducing advanced models.

Classification

Naive baseline: Always predict "on-time" → 57.7% accuracy

Primary metric: F1 & ROC-AUC. Accuracy is misleading at 57.7% baseline. F1 balances precision and recall; ROC-AUC is threshold-independent.

Why F1 over accuracy? Under 42.3% positive rate, a model scoring 60% accuracy is barely above naive — F1 reveals true performance.

Regression

Naive baseline: Always predict mean delay (10.57 min) → MAE ~13.5 min

Primary: MAE & RMSE. MAE is interpretable in minutes. RMSE penalizes large errors (severe delays). R² as secondary.

Why both? A model with MAE 8 min but RMSE 25 min is good on average but fails on severe delays. RMSE catches this.

Evaluation protocol: Time-based split (chronological, no shuffle) — prevents temporal leakage. Ablation study: same algorithm ± weather features to answer RQ2.

Key Design Decisions

6 min
Delay threshold

Aligned with DB's "significant delay." 5 min too noisy; 15 min too few positives.

Time split
Train/test split

Not random — random leaks future into past. Chronological split reflects real deployment.

July only
Scope

Proof-of-concept shard for fast iteration. Pipeline parameterized for any month.

ICE only
Train type

High-profile long-distance. Removing other types avoids confounding from different operational profiles.

Planned
Time used for merge

departure_planned_time (not actual) to avoid leakage into features.

Left join
Merge type

Preserves all ICE rows. Inner join would silently drop 14K rows without weather.

7 — Code Review

How the code is structured and what makes it special.

Pipeline Architecture

5 sequential notebooks, each outputs a Parquet file consumed by the next:

01 Define 02 Acquire 03 Clean 04 Integrate 05 Explore

Each notebook reads config from data/reference/*.json and writes a report JSON on completion. No hardcoded paths or manual steps.

What's Special

  • Config-driven: 21 JSON files define configs, data dictionaries, merge strategies, and reports per notebook — full audit trail without a database.
  • Streaming Parquet: Full DB is 54.8 GB. Notebooks read in 100K-row chunks via PyArrow, filter ICE, and write a 5.2 MB subset.
  • Auto-generated reports: Each notebook writes a structured JSON summary (cleaning stats, merge validation, EDA findings) → reproducible by re-running.
  • Separate presentation: presentation.html lives in its own repo, decoupled from notebooks — enables independent deployment (Netlify).
